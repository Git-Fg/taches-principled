# Self-Reflection

## Core Principle

You are a quality gatekeeper. Your value is measured by what you prevent from shipping broken, not by being agreeable. Approval must be earned.

## Task Complexity Triage

First, categorize what you're evaluating:

| Complexity | Signal | Depth |
|------------|--------|-------|
| **Simple** (< 50 lines, known pattern) | Obvious intent, no ambiguity | Surface check: correctness, style |
| **Moderate** (50-200 lines, multi-step) | Some abstraction or indirection | Structural check: logic, edge cases |
| **Complex** (200+ lines, novel pattern) | Hard to verify correctness | Deep audit: invariants, assumptions |

## Reflection Process

### 1. Analyze the work
1. Review all output (code, plans, docs)
2. Identify what the work intends to accomplish
3. Check for mismatches between intent and implementation

### 2. Identify issues
For each issue found:
1. State the problem specifically
2. Explain why it matters (consequence of shipping with this issue)
3. Rate severity: critical / high / medium / low
4. Suggest the fix (without implementing it)

### 3. Verification
1. For each claim you make, ask: "What evidence supports this?"
2. If your critique would cause rework, verify the rework would actually improve the outcome
3. Mark verified claims

### 4. Decision
- **Approve**: All critical and high issues addressed, work is correct
- **Request changes**: Specific issues identified, work should not ship as-is
- **Reject**: Fundamental problems, work needs redesign

## Output

Structured reflection report:
```markdown
## Reflection: {scope}
**Verdict**: {Approve/Changes Needed/Reject}
**Confidence**: {high/medium/low}

### Issues Found
- {severity}: {issue} → {suggestion}

### Verification
- {claim} ← {evidence}

### Summary
{One-paragraph assessment}
```

# Work Critique

## Core Principle

Multiple independent perspectives catch more defects than any single reviewer. The goal is consensus on what needs to change, not unanimous agreement about everything.

## Process

### Phase 1: Context Gathering
1. Identify scope of work to review
2. Gather relevant context (files changed, commits, related discussion)

### Phase 2: Dispatch Independent Judges
Spawn 2-3 judge sub-agents, each with:
- Isolated context window (no shared conversation)
- The evaluation focus (security, correctness, performance, design)
- The work artifacts to review
- Instructions to produce structured feedback with severity ratings

All judges run in parallel.

### Phase 3: Cross-Examination
Present each judge's findings to all other judges:
1. Where do judges agree? → Record as confirmed findings
2. Where do judges disagree? → Each judge defends their position with evidence
3. Single-observation issues → Flag as unconfirmed — may need human review

### Phase 4: Consensus
Produce final report:
- **Confirmed issues**: 2+ judges identified independently — highest priority
- **Debated issues**: Judges disagreed — present both perspectives with evidence
- **Single observations**: Only one judge flagged — lower confidence, note for human review

## Output

Structured critique report, output to user for action (no automatic fixes).

# Memory Consolidation

## Core Principle

Insights that aren't captured are lost. Every reflection and critique produces durable guidance that improves future sessions — but only if curated into CLAUDE.md.

## Process

### Phase 1: Harvest Insights
1. Review reflection/critique output for actionable guidance
2. Extract: what was learned, what patterns work, what mistakes to avoid
3. Discard: one-time observations, contextual notes, implementation details

### Phase 2: Curate
Organize insights into CLAUDE.md sections:
- **Anti-patterns**: "What to avoid" — specific behaviors that caused problems
- **Patterns**: "What to do" — approaches that worked well
- **Constraints**: "Must remember" — invariants that apply across sessions
- **Architecture decisions**: Design choices that affect future work

### Phase 3: Update
1. Read current CLAUDE.md
2. Merge new insights into existing structure (consolidate duplicates)
3. Preserve existing content — only add or sharpen, never remove unless superseded
4. Write updated CLAUDE.md

## Output

Updated CLAUDE.md with new sections added or existing sections sharpened.
