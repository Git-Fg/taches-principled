# Plan: Consolidate All Agents Into Skills

## Context

**Current state:**
- `agents/` (root) — code-reviewer, prompt-engineer, skill-auditor, subagent-auditor
- `skills/create-plans/agents/` — explorer, researcher, architect, critic, implementer, verifier
- `skills/execute-plans/agents/` — critic (milestone review variant)
- `templates/` — autonomous-execution.md, segment-execution.md (worker prompt structures)

**User intent:** Remove XML structures from agent files, use generalized system prompts inlined within skills.

**Key distinction:**
1. **Shared agents** (`agents/` root) — invoked directly by users, not by skills
2. **Skill-owned agents** (`skills/*/agents/`) — spawned by skills during execution

**Goal:** Consolidate all agent definitions into skills. No more separate `agents/` directory.

## Phase 2: Handle Shared Agents (Root `agents/`)

### 2.1 Analysis of Root Agents

The 4 root agents are **invoked by users directly**, not by skills:

| Agent | Invocation Pattern |
|-------|-------------------|
| code-reviewer | User: "use code-reviewer to review..." |
| prompt-engineer | User: "use prompt-engineer to create..." |
| skill-auditor | User: "use skill-auditor to audit..." |
| subagent-auditor | User: "use subagent-auditor to audit..." |

**Options:**

**Option A: Inline into skill that teaches them**
- code-reviewer → inlined into create-skills (teaches skill authoring, can also teach code review)
- skill-auditor → inlined into create-skills
- subagent-auditor → inlined into create-skills
- prompt-engineer → keep separate or inline into create-prompts/execute-prompts

**Option B: Keep as command-based invocation**
- These agents are user-invoked, not skill-spawned
- They could remain as `/agents` but with generalized prompts

**Option C: Convert to skill-based invocation**
- Create a new skill like `audit-code` that wraps code-reviewer
- User invokes skill, skill delegates to agent
- But this adds indirection without clear benefit

### 2.2 Recommendation

**Option A (Inline into skills):**

1. **skill-auditor + subagent-auditor** → inline into `create-skills` as "audit skills you create" guidance
2. **code-reviewer** → inline into `create-skills` as "review code generated by skills" guidance, OR keep as user-invoked agent
3. **prompt-engineer** → already has `skills:` field referencing create-prompts/execute-prompts; could inline into those skills

**Rationale:** If an agent is only used by users directly, it doesn't need to be inlined. But if it provides patterns that could be useful inline, inlining teaches through example.

When a subagent prompt needs to be portable across projects, shared between skills, or version-controlled separately — store it in `agents/` rather than inlining it in SKILL.md.
